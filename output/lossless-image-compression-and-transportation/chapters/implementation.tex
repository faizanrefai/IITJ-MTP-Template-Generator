\subsubsection{Huffman Coding}


\paragraph{Source Entropy and the Shannon Bound}

Let an image be represented as a discrete source that emits pixel intensity values from an alphabet $\mathcal{A} = \{0, 1, \dots, M-1\}$, where $M = 256$ for 8-bit grayscale images. For a given pixel value $i \in \mathcal{A}$, let $p_i$ denote its empirical probability (i.e., frequency divided by the total number of pixels). The source entropy $H(I)$ is defined as:

\begin{equation}
    H(I) = -\sum_{i=0}^{M-1} p_i \log_{2} p_i \quad \text{(bits per pixel)}
\end{equation}

This quantity represents the theoretical lower bound on the average number of bits per pixel required to encode the image losslessly \cite{VemuriSectionII}. The practical implication is that no lossless compression algorithm can achieve an average bit rate lower than $H(I)$; the degree to which an algorithm approaches this bound serves as a primary measure of its coding efficiency.

\paragraph{Compression Ratio and Bit Rate}

The performance of a lossless compression algorithm is typically quantified by the \textit{compression ratio} ($CR$) or the \textit{bit rate} ($BR$, measured in bits per pixel). For an image of dimensions $R \times C$ originally stored with $b$ bits per pixel, the original size is $R \cdot C \cdot b$ bits. If the compressed representation occupies $S$ bits, then:

\begin{equation}
    CR = \frac{R \cdot C \cdot b}{S}, \quad BR = \frac{S}{R \cdot C}
\end{equation}

The two metrics are related by the simple identity:

\begin{equation}
    CR = \frac{b}{BR}
\end{equation}

A compression ratio greater than $1$ indicates effective compression, while a ratio less than $1$ signifies expansion. The bit rate $BR$ should ideally be close to the source entropy $H(I)$ for an efficient algorithm.


\paragraph{Algorithmic Principle}

Huffman coding, introduced by David A. Huffman in his seminal 1952 paper while a Ph.D. student at MIT, is a greedy algorithm that constructs an optimal prefix code for a given set of symbols and their probabilities \cite[Section IV-A]{VemuriEtAl}. The algorithm operates on the principle that symbols occurring more frequently in the input data are assigned shorter codewords, while less frequent symbols receive longer codewords, thereby minimizing the expected code length.

\vspace{0.5em}\noindent
\textbf{Definition 4.1 (Prefix Code).} A code is a \textit{prefix code} (also called a prefix-free code) if no codeword is a prefix of any other codeword. This property ensures unique decodability without requiring delimiters between codewords.

\vspace{0.5em}\noindent
\textbf{Definition 4.2 (Kraft--McMillan Inequality).} A necessary and sufficient condition for the existence of a prefix code with codeword lengths $l_1, l_2, \dots, l_n$ is that the Kraft--McMillan inequality holds:

\begin{equation}
    \sum_{i=1}^{n} 2^{-l_i} \leq 1
\end{equation}

Huffman codes achieve equality in this inequality when the code is complete \cite{reference19}.

\paragraph{The Huffman Algorithm}

Given a set of symbols with associated probabilities (frequencies), the Huffman algorithm proceeds as follows:

\begin{description}
    \item[Frequency Analysis:] Compute the frequency of each pixel intensity value in the image.
    \item[Tree Construction:] Create a leaf node for each symbol with its probability. While more than one node remains:
    \begin{itemize}
        \item Extract the two nodes with the smallest probabilities from a min-priority queue.
        \item Create a new internal node whose probability is the sum of these two probabilities.
        \item Set the two extracted nodes as left and right children of the new node.
        \item Insert the new node back into the priority queue.
    \end{itemize}
    \item[Code Assignment:] Traverse the tree from the root to each leaf, assigning `0' for left branches and `1' for right branches. The sequence of bits along each path constitutes the Huffman codeword for the corresponding symbol.
\end{description}

\paragraph{Mathematical Optimality}

The optimality of Huffman codes is formally established through the following theorem.

\vspace{0.5em}\noindent
\textbf{Theorem 4.1 (Huffman Optimality).} For a given source with symbol probabilities $\{p_1, p_2, \dots, p_n\}$, the Huffman algorithm produces a prefix code that minimizes the expected codeword length $L = \sum_{i=1}^{n} p_i l_i$ among all prefix codes \cite{reference20}.

\vspace{0.5em}\noindent
\textit{Proof Sketch.} The proof proceeds by induction on the number of symbols. The key insight is that in an optimal code, the two least probable symbols must have the longest codewords and must differ only in the last bit. Huffman's greedy merging strategy preserves this property at each iteration, leading to global optimality.

\paragraph{Implementation Considerations}

The present implementation employs a min-heap (priority queue) data structure to efficiently retrieve the two smallest probability nodes at each iteration, achieving $\mathcal{O}(n \log n)$ time complexity for constructing the Huffman tree, where $n$ is the number of distinct pixel values in the image. The encoding phase then produces a binary string by traversing the tree for each pixel.

\paragraph{Empirical Observations.} For the $512 \times 512$ Lenna image with $215$ distinct pixel values, Huffman coding achieved a compression ratio of $1.071:1$, corresponding to an average bit rate of approximately $7.47$ bits per pixel (slightly above the empirical entropy of $7.445$ bits per pixel). This near-entropy performance demonstrates the theoretical optimality of the Huffman construction.

However, Huffman coding exhibits two fundamental limitations \cite{reference9}:
\begin{itemize}
    \item Each symbol must be encoded with an integer number of bits, even when the symbol's self-information is fractional.
    \item The code must be transmitted alongside the compressed data, incurring overhead that becomes significant for small alphabets.
\end{itemize}

These limitations motivate the development of arithmetic coding, which overcomes the per-symbol integer bit constraint.

\paragraph{Huffman Coding Implementation}

The core compression logic for the Huffman algorithm is implemented in Python. The implementation utilizes a min-heap priority queue to efficiently construct the optimal prefix tree, followed by a Depth-First Search (DFS) traversal to assign the binary codewords.

\begin{verbatim}
import heapq
from collections import Counter

class HuffmanCoding:
    def compress(self, image_array):
        # 1. Build frequency dictionary using standard library
        frequency = dict(Counter(image_array.flatten()))
        
        # 2. Build Huffman tree using min-heap
        heap = [HeapNode(value, freq) for value, freq in frequency.items()]
        heapq.heapify(heap)
        
        while len(heap) > 1:
            node1 = heapq.heappop(heap)
            node2 = heapq.heappop(heap)
            merged = HeapNode(None, node1.freq + node2.freq)
            merged.left, merged.right = node1, node2
            heapq.heappush(heap, merged)
            
        # 3. Generate codes via DFS traversal
        codes = {}
        def generate_codes(node, current_code):
            if node.pixel_value is not None:
                codes[node.pixel_value] = current_code
                return
            generate_codes(node.left, current_code + "0")
            generate_codes(node.right, current_code + "1")
            
        generate_codes(heap[0], "")
        
        # 4. Encode image
        encoded_bits = ''.join(codes[p] for p in image_array.flatten())
        original_bits = image_array.size * 8
        compression_ratio = original_bits / len(encoded_bits)
        
        return encoded_bits, compression_ratio, codes
\end{verbatim}

\vspace{1em}
\noindent\textbf{Time Complexity.} The time complexity is $\mathcal{O}(n \log n)$ for the heap operations, where $n$ is the number of distinct pixel values. For standard 8-bit images, $n \leq 256$, making the time complexity for tree construction effectively constant. The dominant factor in execution time becomes the linear $\mathcal{O}(N)$ traversal required to encode the $N$-pixel image array.

\subsubsection{Arithmetic Coding: Approaching the Entropy Bound}


\paragraph{Theoretical Foundation}

Arithmetic coding represents a significant advancement over Huffman coding by encoding an entire message as a single real number in the interval $[0, 1)$, rather than assigning discrete codewords to individual symbols \cite{VemuriSectionIVC}. This approach enables the encoding of fractional bits per symbol, allowing the code length to approach the entropy bound arbitrarily closely.

The fundamental insight is that any message can be represented by a subinterval of $[0, 1)$, where the length of the subinterval equals the product of the probabilities of the symbols in the message. The encoder outputs a binary representation of any number within this final interval.

\paragraph{Encoding Algorithm}

Given an alphabet $\mathcal{A} = \{a_1, a_2, \dots, a_m\}$ with a cumulative probability distribution $F(a_i) = \sum_{j=1}^{i} p(a_j)$, where $F(a_0) = 0$ and $F(a_m) = 1$, the arithmetic encoding process proceeds as follows \cite{reference11}:

\begin{enumerate}
    \item Let $[L, H)$ denote the current interval, initialized to $[0, 1)$.
    \item For each symbol $x_i$ in the input sequence:
    \begin{itemize}
        \item Compute the width of the current interval: $w = H - L$
        \item Partition $[L, H)$ into subintervals proportional to the symbol probabilities. For symbol $x_i$, the new interval is mathematically defined as:
        \begin{equation}
            [L + w \cdot F(x_{i-1}), L + w \cdot F(x_i))
        \end{equation}
        \item Replace $[L, H)$ with the subinterval corresponding to the encoded symbol.
        \item Output bits as they become known through a renormalization process.
    \end{itemize}
    \item After processing all symbols, any number within the final interval can be selected as the codeword. In practice, a binary fraction with sufficient precision is chosen.
\end{enumerate}

\vspace{0.5em}\noindent
\textbf{Renormalization.} To prevent the interval width from becoming too small for finite-precision arithmetic, a renormalization procedure scales the interval whenever it falls below a threshold (typically $0.25$) and outputs the most significant bits of $L$ and $H$ as they become identical \cite{reference6}.

\paragraph{Decoding Algorithm}

The decoding process reconstructs the original message by iteratively determining which symbol's interval contains the encoded value $v$, then narrowing the interval accordingly:

\begin{enumerate}
    \item Initialize $[L, H) = [0, 1)$.
    \item While symbols remain to be decoded:
    \begin{itemize}
        \item Find the symbol $x_i$ such that:
        \begin{equation}
            L + (H-L) \cdot F(x_{i-1}) \leq v < L + (H-L) \cdot F(x_i)
        \end{equation}
        \item Output $x_i$.
        \item Narrow the interval to $[L + (H-L) \cdot F(x_{i-1}), L + (H-L) \cdot F(x_i))$.
        \item Apply the same renormalization procedure as the encoder to maintain precision.
    \end{itemize}
\end{enumerate}

\paragraph{Optimality and Advantages}

\vspace{0.5em}\noindent
\textbf{Theorem 4.2 (Arithmetic Coding Optimality).} For a memoryless source, the average length of an arithmetic code equals the source entropy $H$, measured in bits per symbol, asymptotically as the message length approaches infinity \cite{reference9}.

\vspace{0.5em}
Unlike Huffman coding, which requires an integer number of bits per symbol, arithmetic coding can encode fractional bits by distributing them across symbol boundaries. This capability yields two significant advantages:
\begin{itemize}
    \item \textbf{Fractional Bit Encoding:} Symbols with probabilities close to $1$ can be encoded using fewer than $1$ bit on average.
    \item \textbf{Adaptive Modeling:} The probability model can be updated dynamically during encoding, enabling adaptation to non-stationary sources.
\end{itemize}

\paragraph{Implementation Challenges and Results}

The primary challenge in implementing arithmetic coding lies in managing finite-precision arithmetic while maintaining correctness. The present implementation uses 16-bit integer arithmetic with careful renormalization to avoid underflow. The encoding process maintains three primary state variables:
\begin{itemize}
    \item \texttt{low}: the lower bound of the current interval.
    \item \texttt{high}: the upper bound of the current interval.
    \item \texttt{underflow\_bits}: a counter for bits that cannot yet be emitted due to ambiguous most-significant bits.
\end{itemize}

\paragraph{Arithmetic Coding Implementation}\mbox{}\\

The arithmetic coder uses 16-bit integer arithmetic with renormalization to avoid floating-point precision issues.

\begin{verbatim}
from collections import Counter

class ArithmeticCodingLossless:
    def compress(self, image_array):
        pixels = image_array.flatten()
        
        # Build frequency model
        frequencies = dict(Counter(pixels))
        total = len(pixels)
        
        # Build cumulative frequencies
        cumulative = {}
        cum = 0
        for pixel in sorted(frequencies.keys()):
            cumulative[pixel] = (cum, cum + frequencies[pixel])
            cum += frequencies[pixel]
            
        # Initialize interval [low, high)
        low, high = 0, (1 << 16) - 1
        underflow = 0
        output_bits = []
        
        for pixel in pixels:
            range_width = high - low + 1
            cum_low, cum_high = cumulative[pixel]
            
            # Narrow interval
            high = low + (range_width * cum_high) // total - 1
            low = low + (range_width * cum_low) // total
            
            # Renormalization loop
            while True:
                if high < (1 << 15):          # Upper half (0xxx...)
                    output_bits.append(0)
                    output_bits.extend([1] * underflow)
                    underflow = 0
                elif low >= (1 << 15):        # Lower half (1xxx...)
                    output_bits.append(1)
                    output_bits.extend([0] * underflow)
                    underflow = 0
                    low -= (1 << 15)
                    high -= (1 << 15)
                elif low >= (1 << 14) and high < (3 << 14):  # Underflow
                    underflow += 1
                    low -= (1 << 14)
                    high -= (1 << 14)
                else:
                    break
                    
                low <<= 1
                high = (high << 1) | 1
                
        # Flush remaining bits
        underflow += 1
        output_bits.append(0 if low < (1 << 15) else 1)
        output_bits.extend([1 if low < (1 << 15) else 0] * underflow)
        
        # Calculate final metrics
        original_bits = len(pixels) * 8
        compression_ratio = original_bits / len(output_bits)
        metadata = cumulative
        
        return bytes(output_bits), compression_ratio, metadata
\end{verbatim}

\vspace{1em}
\noindent\textbf{Renormalization Complexity Analysis.} Each symbol processed triggers a constant number of renormalization steps on average. The \texttt{while} loop executes at most $\mathcal{O}(\log(1/p_{\min}))$ times per symbol in the worst case, where $p_{\min}$ is the smallest symbol probability. In practice, with 16-bit precision, the loop runs fewer than 5 iterations per symbol for natural images, yielding $\mathcal{O}(N)$ overall time, where $N$ is the number of pixels.

\vspace{1em}
\noindent\textbf{Visual Verification.} The integrity of the compression algorithm was validated through a full-cycle encoding and decoding process. The arithmetic decoder successfully reconstructed the image array from the scaled bitstream. Computation of the absolute difference between the original and reconstructed image arrays yielded a maximum error of zero, confirming that the 16-bit integer approximation with renormalization is strictly lossless.

\subsubsection{LZW Coding: Dictionary-Based Compression}\mbox{}\\

\subparagraph{Algorithmic Framework}\mbox{}\\

Lempel--Ziv--Welch (LZW) coding belongs to the class of dictionary-based compression algorithms, which replace repeated substrings with shorter codes referencing entries in a dynamically constructed dictionary \cite{VemuriSectionIVB}. Unlike Huffman and arithmetic coding, which rely on explicit probability models of individual symbols, LZW adaptively learns patterns from the input data during a single pass.

The algorithm was developed by Abraham Lempel and Jacob Ziv (LZ78, 1978) and subsequently refined by Terry Welch (1984). LZW is widely deployed in standard formats, including the GIF image format, the UNIX \texttt{compress} utility, and the PDF file format \cite{reference18}.

\subparagraph{Encoding Process}\mbox{}\\

Let $\Sigma$ denote the set of all possible symbols (pixel intensity values $0$--$255$). The LZW encoder maintains a dictionary $\mathcal{D}$ that maps strings (sequences of symbols) to fixed-length codes. The encoding procedure follows these steps \cite{reference17}:

\begin{description}
    \item[Dictionary Initialization:] Initialize $\mathcal{D}$ with all single-symbol strings, assigning each symbol a unique code ($0$--$255$ for 8-bit grayscale).
    \item[String Matching:] Initialize the current string $s$ to the first symbol. For each subsequent symbol $c$ in the input:
    \begin{itemize}
        \item If the concatenation $s + c$ exists in $\mathcal{D}$, set $s = s + c$.
        \item Otherwise:
        \begin{itemize}
            \item Output the code for $s$.
            \item Add $s + c$ to $\mathcal{D}$ with the next available code.
            \item Set $s = c$.
        \end{itemize}
    \end{itemize}
    \item[Termination:] After processing all symbols, output the code for the final string $s$.
\end{description}

Formally, let the input be a sequence $x_1, x_2, \dots, x_N$. Define the dictionary as a set of strings with associated codes. For iteration $i$, maintain a prefix $w$. When reading the next symbol $c$, if $wc \in \mathcal{D}$, extend $w \leftarrow wc$; otherwise, output $\text{code}(w)$, add $wc$ to $\mathcal{D}$, and reset $w \leftarrow c$.

\subparagraph{Decoding Process}\mbox{}\\

A remarkable property of LZW is that the decoder can reconstruct both the original data and the dictionary simultaneously without receiving the dictionary explicitly \cite{reference18}. The decoder processes the compressed code sequence, reconstructing strings by:

\begin{description}
    \item[Initialization:] Initialize the same base dictionary as the encoder ($0$--$255$ mapping to single symbols).
    \item[Decoding Loop:] For each input code $k$:
    \begin{itemize}
        \item If $k$ is in the dictionary, output the corresponding string $str$.
        \item If $k$ equals the next code to be added (a special edge case), set $str = prev + \text{first\_char}(prev)$.
        \item Add $prev + \text{first\_char}(str)$ to the dictionary.
        \item Set $prev = str$.
    \end{itemize}
\end{description}

The special case occurs when the encoder would have added a string that the decoder has not yet encountered---a situation that arises for certain repeating patterns (e.g., character sequences like $cScSc$). The decoder elegantly resolves it by recognizing that the extension character must be the first character of the previous output.

\subparagraph{Complexity and Performance}\mbox{}\\

LZW achieves $\mathcal{O}(N)$ time complexity for both encoding and decoding, with dictionary operations (lookup and insertion) amortized to constant time using hash-based data structures. The primary performance characteristic is the growth of the dictionary during processing: each new string encountered adds an entry, and the dictionary size is bounded by the available code space (e.g., $2^{12} = 4096$ entries for standard 12-bit codes, though modern implementations allow dynamic resizing).

\vspace{1em}
\noindent\textbf{Experimental Results and Analysis}\mbox{}\\

To evaluate the practical performance of LZW coding, the algorithm was tested on the same $512 \times 512$ 8-bit grayscale Lenna image used for prior evaluations. 

\vspace{1em}
\noindent\textbf{Quantitative Metrics.} LZW achieved a remarkable \textbf{compression ratio of 1.528:1}, significantly outperforming both Huffman ($1.071:1$) and arithmetic coding ($1.070:1$) on this specific dataset. Based on this ratio, the effective bit rate of the encoded image is:

\begin{equation}
    BR_{\text{LZW}} = \frac{8}{1.528} \approx 5.235 \text{ bits per pixel}
\end{equation}

This superior performance stems from LZW's ability to exploit repetitive pixel patterns and spatial correlations inherent in natural images. Huffman and arithmetic coding, as purely symbol-wise memoryless entropy coders, cannot directly capture these higher-order redundancies without explicit multi-symbol probability modeling.

\vspace{1em}
\noindent\textbf{Dictionary and Data Analysis.} During the single-pass encoding, the dynamic dictionary expanded to contain $114,660$ entries, indicating substantial pattern repetition within the image data. The final compressed data comprised $114,405$ codes. Assuming a standard 12-bit code width, this represents a reduction from the original $2,097,152$ bits to approximately $1,372,860$ bits.

\vspace{1em}
\noindent\textbf{Visual Verification.} As with the entropy coders, LZW is strictly mathematically lossless. The decoding phase successfully regenerated the complete dictionary on the fly and reconstructed the exact image array, verified by an absolute error map showing zero deviation across all $262,144$ pixels.

\subsubsection{Spatial Domain}

\textbf{What are Spatial Domain Algorithms?}

Instead of transforming the image into frequencies (like JPEG's Discrete Cosine Transform), spatial domain methods work directly on the pixel values themselves.

\paragraph{The Two-Stage Process}

\paragraph{Stage 1: Pre-processing}\mbox{}\\
The image is broken down into regions or simplified using one of two approaches:

    \centering
    \begin{tabular}{|l|p{4cm}|p{4.5cm}|p{3.5cm}|}
        \hline
        \textbf{Approach} & \textbf{What it does} & \textbf{Example} & \textbf{Storage cost} \\
        \hline
        \textbf{Regular shapes} & Split image into fixed grids (rectangles). & $8 \times 8$ blocks like JPEG. & \textbf{Low} -- just need block coordinates. \\
        \hline
        \textbf{Irregular shapes} & Segment based on image content. & Split sky region, then tree region separately. & \textbf{High} -- need to store complex boundaries. \\
        \hline
    \end{tabular}
    \caption{Comparison of shape segmentation approaches in spatial domain algorithms.}
    \label{tab:shape_segmentation}
\end{table}

\noindent\textbf{Visual Example:}

    \includegraphics[width=1.0\textwidth]{spatial.png}

\paragraph{Stage 2: Encoding}\mbox{}\\
After segmentation, each region is encoded efficiently based on its visual characteristics:
\begin{itemize}
    \item \textbf{Smooth regions} $\rightarrow$ store the average color.
    \item \textbf{Textured regions} $\rightarrow$ store pattern information.
    \item \textbf{Edges} $\rightarrow$ store boundary information.
\end{itemize}

\subsubsection*{Why Regular Shapes Win}

In practical implementation, \textbf{regular shapes (rectangles) are typically preferred} over irregular content-based shapes because:
\begin{itemize}
    \item \textbf{Simple to store:} The algorithm only needs to record position and size.
    \item \textbf{Less overhead:} There is no need to encode and transmit complex mathematical boundary data.
    \item \textbf{Faster processing:} Fixed grids allow for highly predictable, parallelized computational patterns.
\end{itemize}
