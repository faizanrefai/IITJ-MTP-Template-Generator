\subsubsection{Huffman Coding}


\paragraph{Source Entropy and the Shannon Bound}

An image is represented heavily as a discrete source by which pixel intensity values are actively emitted. These values are pulled from an alphabet $\mathcal{A} = \{0, 1, \dots, M-1\}$, in which $M = 256$ is utilized for standard 8bit grayscale images. For any given pixel value $i \in \mathcal{A}$, its empirical probability is denoted entirely by $p_i$ (calculated strictly as the frequency divided by the absolute total number of pixels). Source entropy $H(I)$ is defined clearly below:

\begin{equation}
    H(I) = -\sum_{i=0}^{M-1} p_i \log_{2} p_i \quad \text{(bits per pixel)}
\end{equation}

The theoretical lower bound regarding the average number of bits per pixel that are strictly required to encode the image losslessly is represented completely by this quantity \cite{Salomon1997}. A strict practical implication is created by this. An average bit rate lower than $H(I)$ cannot possibly be achieved by any lossless compression algorithm. Coding efficiency is measured primarily by looking directly at the exact degree to which an algorithm approaches this specific bound.

\paragraph{Compression Ratio and Bit Rate}

The raw performance of a lossless compression algorithm is quantified typically by looking at the \textit{compression ratio} ($CR$) or the \textit{bit rate} ($BR$, measured strictly in bits per pixel). When observing an image containing dimensions $R \times C$ that was stored originally with $b$ bits per pixel, $R \cdot C \cdot b$ bits is calculated as the original size. If $S$ bits are completely occupied by the compressed representation, the following is established:

\begin{equation}
    CR = \frac{R \cdot C \cdot b}{S}, \quad BR = \frac{S}{R \cdot C}
\end{equation}

A highly simple identity relates the two metrics together:

\begin{equation}
    CR = \frac{b}{BR}
\end{equation}

Effective compression is indicated strongly when a compression ratio greater than $1$ is produced, whereas expansion is signified entirely if a ratio less than $1$ is observed. To ensure an efficient algorithm is achieved, the bit rate $BR$ should ideally sit very close to the source entropy $H(I)$.


\paragraph{Algorithmic Principle}

Huffman coding was introduced fully by David A. Huffman back in his seminal 1952 paper, which was written while he was still an active Ph.D. student over at MIT \cite{Huffman1952}. An optimal prefix code is constructed completely for a given set of symbols alongside their probabilities by this highly greedy algorithm \cite[Section IV-A]{Salomon1997}. A core principle is operated upon by the algorithm. Shorter codewords are assigned directly to symbols that occur significantly more frequently within the input data. Longer codewords are subsequently received by symbols that appear much less frequently. The expected code length is minimized entirely by doing this. \cite{Huffman1952}

\vspace{0.5em}\noindent
\textbf{Definition 4.1 (Prefix Code).} A code is classified strictly as a \textit{prefix code} (referred to also as a prefixfree code) if it is ensured that no codeword acts as a prefix for any other available codeword. Unique decodability is ensured completely by this specific property. Delimiters are not required at all between the codewords.

\vspace{0.5em}\noindent
\textbf{Definition 4.2 (Kraft--McMillan Inequality).} To guarantee the existence of a prefix code containing specific codeword lengths of $l_1, l_2, \dots, l_n$, a necessary and entirely sufficient condition is demanded. The Kraft--McMillan inequality must hold true:

\begin{equation}
    \sum_{i=1}^{n} 2^{-l_i} \leq 1 
\end{equation}

Equality within this specific inequality is achieved perfectly by Huffman codes whenever the code itself is rendered complete \cite{Huffman1952}.

\paragraph{The Huffman Algorithm}

When a set of symbols paired closely with associated probabilities (frequencies) is provided, the Huffman algorithm proceeds directly through the steps listed below:

\begin{description}
    \item[Frequency Analysis:] The exact frequency of every single pixel intensity value contained within the image is computed heavily.
    \item[Tree Construction:] A dedicated leaf node is created for each individual symbol alongside its matched probability. As long as more than one node remains fully available:
    \begin{itemize}
        \item The two nodes possessing the absolute smallest probabilities are extracted directly out of a minpriority queue.
        \item A completely new internal node is created. The combined sum of these two probabilities serves exactly as its assigned probability.
        \item The two initially extracted nodes are set actively as the left and right children belonging to the new node.
        \item The new node is inserted straight back into the priority queue.
    \end{itemize}
    \item[Code Assignment:] The tree is traversed fully, moving straight from the root down to each individual leaf. A value of `0' is assigned strictly for left branches while a `1' is assigned for right branches. The sequence of bits collected along each isolated path constitutes the finalized Huffman codeword utilized for the corresponding symbol.
\end{description}

\paragraph{Mathematical Optimality}

Formal establishment of the sheer optimality possessed by Huffman codes is achieved completely through the theorem detailed below.

\vspace{0.5em}\noindent
\textbf{Theorem 4.1 (Huffman Optimality).} When dealing with a given source containing strict symbol probabilities $\{p_1, p_2, \dots, p_n\}$, a specific prefix code is produced completely by the Huffman algorithm. The expected codeword length $L = \sum_{i=1}^{n} p_i l_i$ is minimized entirely among all possible prefix codes by this output \cite{Huffman1952}.

\vspace{0.5em}\noindent
\textit{Proof Sketch.} Induction resting upon the total number of symbols is utilized to drive the proof forward. A key insight is highlighted heavily here. Within an optimal code, the two absolutely least probable symbols are required to have the longest codewords. Furthermore, they must differ exclusively by the final bit. This specific property is preserved completely at every single iteration by Huffman's greedy merging strategy. Global optimality is reached entirely as a result.

\paragraph{Implementation Considerations}

A minheap (priority queue) data structure is employed actively by the present implementation. This allows the two smallest probability nodes to be retrieved with massive efficiency during each iteration. A strict time complexity of $\mathcal{O}(n \log n)$ is achieved completely for constructing the Huffman tree, with $n$ representing the absolute number of distinct pixel values held in the image. A binary string is subsequently produced during the encoding phase. This happens as the tree is traversed completely for every single pixel.

\paragraph{Empirical Observations.} The $512 \times 512$ Lenna image, possessing exactly $215$ distinct pixel values, was tested. A compression ratio sitting at $1.071:1$ was achieved completely by the Huffman coding setup. An average bit rate roughly equaling $7.47$ bits per pixel was observed (resting slightly above the empirical entropy bound of $7.445$ bits per pixel). The sheer theoretical optimality tied to the Huffman construction is demonstrated perfectly by this nearentropy performance.

Two heavily fundamental limitations are exhibited by Huffman coding, however \cite{Huffman1952}:
\begin{itemize}
    \item An integer number of bits must be utilized to encode every single symbol. This rule remains strict even whenever the selfinformation of the symbol itself is highly fractional.
    \item The code is required to be transmitted directly alongside the completely compressed data. Heavy overhead is incurred by this action, becoming massively significant when dealing with small alphabets.
\end{itemize}

The active development of arithmetic coding is motivated entirely by these massive limitations. The strict persymbol integer bit constraint is completely overcome by arithmetic methods.

\paragraph{Huffman Coding Implementation}

Python is utilized heavily to implement the core compression logic required for the Huffman algorithm. The optimal prefix tree is constructed massively efficiently by utilizing a strictly defined minheap priority queue within the implementation. The exact assignment of the binary codewords is handled directly by a subsequent DepthFirst Search (DFS) traversal.

\begin{verbatim}
import heapq
from collections import Counter

class HuffmanCoding:
    def compress(self, image_array):
        # 1. Build frequency dictionary using standard library
        frequency = dict(Counter(image_array.flatten()))
        
        # 2. Build Huffman tree using min-heap
        heap = [HeapNode(value, freq) for value, freq in frequency.items()]
        heapq.heapify(heap)
        
        while len(heap) > 1:
            node1 = heapq.heappop(heap)
            node2 = heapq.heappop(heap)
            merged = HeapNode(None, node1.freq + node2.freq)
            merged.left, merged.right = node1, node2
            heapq.heappush(heap, merged)
            
        # 3. Generate codes via DFS traversal
        codes = {}
        def generate_codes(node, current_code):
            if node.pixel_value is not None:
                codes[node.pixel_value] = current_code
                return
            generate_codes(node.left, current_code + "0")
            generate_codes(node.right, current_code + "1")
            
        generate_codes(heap[0], "")
        
        # 4. Encode image
        encoded_bits = ''.join(codes[p] for p in image_array.flatten())
        original_bits = image_array.size * 8
        compression_ratio = original_bits / len(encoded_bits)
        
        return encoded_bits, compression_ratio, codes
\end{verbatim}

\noindent\textbf{Time Complexity.} For the required heap operations, a strict time complexity of $\mathcal{O}(n \log n)$ is completely established. The number of strictly distinct pixel values is represented by $n$. For purely standard 8-bit images, $n \leq 256$ is completely guaranteed. The actual time complexity tied to tree construction is rendered effectively constant by this strict bounding. The absolute dominant factor driving the total execution time becomes the heavily linear $\mathcal{O}(N)$ traversal. This traversal is strictly required so that the massive $N$-pixel image array can be fully encoded.

\subsubsection{Arithmetic Coding: Approaching the Entropy Bound}


\paragraph{Theoretical Foundation}

A massive advancement over standard Huffman coding is completely represented by arithmetic coding. An entire message is completely encoded purely as a single real number falling directly within the strict interval of $[0, 1)$ rather than discrete codewords being assigned directly to isolated symbols \cite{Salomon1997}. The direct encoding of highly fractional bits per symbol is entirely enabled by this specific approach. Approaching the absolute entropy bound arbitrarily closely is entirely permitted by this code length dynamic.

That any given message can be completely represented strictly by a subinterval of $[0, 1)$ serves as the fundamental insight here. The product of the strict probabilities tied directly to the symbols within the message is equal entirely to the length of this generated subinterval. A binary representation detailing any number housed within this final interval is subsequently outputted completely by the encoder.

\paragraph{Encoding Algorithm}

Given an alphabet $\mathcal{A} = \{a_1, a_2, \dots, a_m\}$ alongside a cumulative probability distribution strictly defined as $F(a_i) = \sum_{j=1}^{i} p(a_j)$ (where $F(a_0) = 0$ and $F(a_m) = 1$ are firmly established), the arithmetic encoding process is executed directly as follows \cite{Witten1987}:

\begin{enumerate}
    \item The current interval is denoted completely by $[L, H)$. Strict initialization to $[0, 1)$ is heavily performed.
    \item For every single symbol $x_i$ present entirely within the input sequence:
    \begin{itemize}
        \item The exact width of the active interval is heavily computed: $w = H - L$
        \item Partitioning of the active $[L, H)$ into strict subintervals is heavily executed. This is strictly proportional to the raw symbol probabilities. The entirely new interval generated directly for symbol $x_i$ is defined mathematically below:
        \begin{equation}
            [L + w \cdot F(x_{i-1}), L + w \cdot F(x_i))
        \end{equation}
        \item The active $[L, H)$ is replaced entirely by the strict subinterval matched directly to the encoded symbol.
        \item Bits are actively outputted completely as they become heavily known via a strict renormalization process.
    \end{itemize}
    \item After every single symbol is completely processed, the selection of any number trapped within the final interval can be performed directly as the ultimate codeword. A pure binary fraction containing highly sufficient precision is completely chosen in heavy practice.
\end{enumerate}

\vspace{0.5em}\noindent
\textbf{Renormalization.} The absolute interval width must be completely prevented from becoming far too small for finiteprecision arithmetic. A strict renormalization procedure is utilized actively. The interval is actively scaled entirely whenever it manages to fall below a tight threshold (typically sitting around $0.25$). The mostsignificant bits belonging directly to $L$ and $H$ are actively outputted completely as soon as they become entirely identical \cite{Witten1987}.

\paragraph{Decoding Algorithm}

The original message is reconstructed heavily by the decoding process. This is achieved completely by iteratively determining exactly which specific symbol's interval completely houses the encoded value $v$. The active interval is then narrowed down heavily accordingly:

\begin{enumerate}
    \item $[L, H) = [0, 1)$ is heavily initialized.
    \item While remaining symbols are actively waiting to be decoded:
    \begin{itemize}
        \item The specific symbol $x_i$ is actively hunted down strictly such that:
        \begin{equation}
            L + (H-L) \cdot F(x_{i-1}) \leq v < L + (H-L) \cdot F(x_i)
        \end{equation}
        \item $x_i$ is completely outputted.
        \item The active interval is narrowed forcefully down strictly to $[L + (H-L) \cdot F(x_{i-1}), L + (H-L) \cdot F(x_i))$.
        \item The precise renormalization procedure utilized previously by the pure encoder is actively applied to strictly maintain tight precision.
    \end{itemize}
\end{enumerate}

\paragraph{Optimality and Advantages}

\vspace{0.5em}\noindent
\textbf{Theorem 4.2 (Arithmetic Coding Optimality).} When heavily analyzing a memoryless source, the exact average length belonging directly to an arithmetic code equals the source entropy $H$ completely. This is measured directly in bits per symbol asymptotically purely as the total message length approaches absolute infinity \cite{Huffman1952}.

\vspace{0.5em}
Standard Huffman coding requires a strict integer number of bits to be heavily assigned per symbol. Arithmetic coding differs completely by actively encoding fractional bits. Distributing them massively across direct symbol boundaries entirely achieves this. Two massively significant advantages are strictly yielded completely by this highly specific capability:
\begin{itemize}
    \item \textbf{Fractional Bit Encoding:} Fewer than $1$ absolute bit on average can be utilized strictly to encode symbols possessing probabilities sitting highly close to $1$.
    \item \textbf{Adaptive Modeling:} Updating the pure probability model dynamically while active encoding occurs is entirely permitted. Heavy adaptation strictly to nonstationary sources is massively enabled by this.
\end{itemize}

\paragraph{Implementation Challenges and Results}

The complete management of finiteprecision arithmetic while strictly maintaining mathematical correctness serves completely as the massive primary challenge encountered when implementing pure arithmetic coding. Strict 16-bit integer arithmetic is utilized heavily by the present implementation. Extreme care via strict renormalization is actively deployed to completely avoid nasty underflow. Three strictly primary state variables are heavily maintained completely by the pure encoding process:
\begin{itemize}
    \item \texttt{low}: the absolute lower bound assigned directly to the active interval.

from collections import Counter

class ArithmeticCodingLossless:
    def compress(self, image_array):
        pixels = image_array.flatten()
        
        # Build frequency model
        frequencies = dict(Counter(pixels))
        total = len(pixels)
        
        # Build cumulative frequencies
        cumulative = {}
        cum = 0
        for pixel in sorted(frequencies.keys()):
            cumulative[pixel] = (cum, cum + frequencies[pixel])
            cum += frequencies[pixel]
            
        # Initialize interval [low, high)
        low, high = 0, (1 << 16) - 1
        underflow = 0
        output_bits = []
        
        for pixel in pixels:
            range_width = high - low + 1
            cum_low, cum_high = cumulative[pixel]
            
            # Narrow interval
            high = low + (range_width * cum_high) // total - 1
            low = low + (range_width * cum_low) // total
            
            # Renormalization loop
            while True:
                if high < (1 << 15):          # Upper half (0xxx...)
                    output_bits.append(0)
                    output_bits.extend([1] * underflow)
                    underflow = 0
                elif low >= (1 << 15):        # Lower half (1xxx...)
                    output_bits.append(1)
                    output_bits.extend([0] * underflow)
                    underflow = 0
                    low -= (1 << 15)
                    high -= (1 << 15)
                elif low >= (1 << 14) and high < (3 << 14):  # Underflow
                    underflow += 1
                    low -= (1 << 14)
                    high -= (1 << 14)
                else:
                    break
                    
                low <<= 1
                high = (high << 1) | 1
                
        # Flush remaining bits
        underflow += 1
        output_bits.append(0 if low < (1 << 15) else 1)
        output_bits.extend([1 if low < (1 << 15) else 0] * underflow)
        
        # Calculate final metrics
        original_bits = len(pixels) * 8
        compression_ratio = original_bits / len(output_bits)
        metadata = cumulative
        
        return bytes(output_bits), compression_ratio, metadata
\end{verbatim}

\subsubsection{LZW Coding: DictionaryBased Compression}\mbox{}\\

\paragraph{Algorithmic Framework}\mbox{}\\

The class of dictionarybased compression algorithms is exactly where LempelZivWelch (LZW) coding completely belongs. Repeated substrings are actively replaced by shorter codes that reference entries housed within a dynamically constructed dictionary by these algorithms \cite{Salomon1997}. Explicit probability models completely tied to individual symbols are heavily relied upon by standard Huffman and arithmetic coding. LZW differs completely by adaptively learning massive patterns strictly from the input data during one singlepass execution. 

The algorithm was actively developed entirely by Abraham Lempel and Jacob Ziv (LZ78) \cite{Ziv1978}. Subsequent refinement was heavily performed later by Terry Welch (1984) \cite{Welch1984}. Wide deployment across standard formats is completely enjoyed by LZW, heavily including the GIF image format, the UNIX \texttt{compress} utility, and the standard PDF file format \cite{Welch1984}.

\paragraph{Encoding Process}\mbox{}\\
The steps listed completely below are actively followed by the strict encoding procedure \cite{Salomon1997}:
The strict set of all possible symbols (pixel intensity values falling strictly between $0$--$255$) is completely denoted by $\Sigma$. A massive dictionary $\mathcal{D}$ is actively maintained directly by the LZW encoder. Strings (sequences of raw symbols) are mapped heavily to fixedlength codes entirely by this dictionary. The steps listed completely below are actively followed by the strict encoding procedure \cite{Welch1984}:

\begin{description}
    \item[Dictionary Initialization:] $\mathcal{D}$ is completely initialized utilizing every single singlesymbol string. A highly unique code ($0$--$255$ specifically for 8-bit grayscale) is actively assigned to every single symbol.
    \item[String Matching:] The strictly current string $s$ is heavily initialized directly to the absolute first symbol. For every single subsequent symbol $c$ found actively within the input:
    \begin{itemize}
        \item If the direct concatenation $s + c$ is found to exist completely within $\mathcal{D}$, $s = s + c$ is strictly set.
        \item Otherwise:
        \begin{itemize}
            \item The exact code matched to $s$ is actively outputted.
            \item $s + c$ is actively added completely to $\mathcal{D}$ alongside the highly next available code.
            \item $s = c$ is strictly set.
        \end{itemize}
    \end{itemize}
    \item[Termination:] The strict code belonging entirely to the final string $s$ is actively outputted completely after every single symbol is heavily processed.
\end{description}

Formally, the input is strictly defined as a massive sequence $x_1, x_2, \dots, x_N$. The dictionary is actively defined completely as a massive set of strings paired directly with associated codes. A strict prefix $w$ is completely maintained specifically for iteration $i$. When the exact next symbol $c$ is actively read, $w \leftarrow wc$ is heavily extended strictly if $wc \in \mathcal{D}$. Otherwise, $\text{code}(w)$ is heavily outputted, $wc$ is completely added directly to $\mathcal{D}$, and $w \leftarrow c$ is strictly reset.

\paragraph{Decoding Process}\mbox{}\\

A highly remarkable property is completely possessed by LZW. Both the purely original data and the massive dictionary can be reconstructed heavily and entirely simultaneously directly by the decoder. The dictionary never needs to be received explicitly for this to happen \cite{Welch1984}. The highly compressed code sequence is heavily processed entirely by the decoder. Strings are completely reconstructed by heavily following these steps:
 \cite{Salomon1997}
\begin{description}
    \item[Initialization:] The exact same base dictionary utilized previously by the encoder ($0$--$255$ mapping directly to absolute single symbols) is heavily initialized.
    \item[Decoding Loop:] For every single active input code $k$:
    \begin{itemize}
        \item The strictly corresponding string $str$ is actively outputted completely if $k$ is found trapped entirely within the dictionary.
        \item $str = prev + \text{first\_char}(prev)$ is strictly set entirely if $k$ is perfectly equal completely to the absolute next code waiting heavily to be added (acting entirely as a highly special edge case).
        \item $prev + \text{first\_char}(str)$ is heavily added completely directly to the active dictionary.
        \item $prev = str$ is strictly set.
    \end{itemize}
\end{description}

This highly special case actively occurs strictly when a string that the decoder has not yet completely encountered would have been heavily added purely by the encoder. This highly specific situation completely arises primarily for certain repeating patterns (e.g., character sequences looking exactly like $cScSc$). It is elegantly resolved entirely by the decoder. It is actively recognized completely that the highly specific extension character must absolutely be the precise first character belonging completely to the previous output.

Absolute $\mathcal{O}(N)$ time complexity is successfully achieved completely by LZW strictly for both pure encoding and active decoding. Dictionary operations (massive lookup and insertion) are heavily amortized entirely to constant time by strictly utilizing hashbased data structures. The massive growth experienced completely by the dictionary during active processing serves heavily as the absolute primary performance characteristic. A totally new entry is actively added completely whenever a totally new string is actively encountered.
